# MBTA Diff Re-Audit Plan

## Scope

Review only the current uncommitted diff under `mbta/`, re-read surrounding logic, and determine whether the prior bug fixes are valid or introduced regressions. No code changes are made in this plan.

Current diff is limited to `mbta/app.js`:

1. `popupInfoRow()` changed `class="${escapeHtml(className)}"` to `class="${className}"`.
2. `renderAlerts()` removed unused `lifecycleAttr` and no longer writes `data-lifecycle` on individual `.alert-item` elements.
3. `animateVehicleTo()` now includes an interpolated `headingDeg` in the per-frame offset object.

## Findings

### 1. `popupInfoRow()` class handling

**Status:** safe with current call sites.

Relevant logic:

- `popupInfoRow(label, value, valueClass = "")` builds `className` from an internal static base class and optional internal caller-provided class.
- Current call sites pass either no `valueClass` or the static string `"popup-time-value"`.
- User/API-derived values remain escaped through `escapeHtml(label)` and `escapeHtml(value)`.

This change does not introduce a new bug under current code because `className` is not user-controlled. It is also visually neutral: emitted classes remain the same for all existing calls.

### 2. `renderAlerts()` removing `.alert-item[data-lifecycle]`

**Status:** safe.

Relevant logic:

- `style.css` lifecycle selectors target only `.alert-indicator[data-lifecycle]`, `.alert-toggle[data-lifecycle]`, and `.route-alert[data-lifecycle]`.
- There are no `.alert-item[data-lifecycle]` selectors.
- The parent elements still receive lifecycle state through:
  - `alertBox.dataset.lifecycle = topLifecycle`
  - `toggleAlertButton.dataset.lifecycle = topLifecycle`
  - `alertIndicator.dataset.lifecycle = topLifecycle`

So removing `data-lifecycle` from individual `.alert-item` rows does not alter styling, alert sorting, alert count, toggle behavior, or service-alert visibility.

### 3. `animateVehicleTo()` heading interpolation

**Status:** partially fixes the intended bug, but introduces a real edge-case regression.

The change fixes the original issue for normal cases: per-frame offsets now include `headingDeg`, so `applyVehicleVisualOffset()` can update `--vehicle-heading` during animation rather than leaving the arrow frozen at the previous heading.

However, the current interpolation is direct numeric interpolation:

```js
headingDeg: Number.isFinite(fromOffset.headingDeg) && Number.isFinite(toOffset.headingDeg)
    ? fromOffset.headingDeg + (toOffset.headingDeg - fromOffset.headingDeg) * eased
    : (toOffset.headingDeg ?? fromOffset.headingDeg)
```

This introduces an angle-wrap bug. For example, interpolating from `350°` to `10°` should move through `0°` by `+20°`, but direct interpolation moves through `180°` by `-340°`. That can make the vehicle heading arrow visibly rotate the long way around during an otherwise small heading change.

This is a new visual regression risk caused by the fix. It does not break data loading or marker animation, but it can affect vehicle arrow visuals around the `-180/180` or `0/360` boundary.

## Recommended Correction

Add a small helper that interpolates angles using the shortest signed angular delta, then use it inside `animateVehicleTo()`.

Suggested helper near the heading utility functions:

```js
function interpolateHeadingDegrees(fromDeg, toDeg, t) {
    if (!Number.isFinite(fromDeg)) return Number.isFinite(toDeg) ? toDeg : undefined;
    if (!Number.isFinite(toDeg)) return fromDeg;
    const delta = ((toDeg - fromDeg + 540) % 360) - 180;
    return fromDeg + delta * t;
}
```

Then replace the inline interpolation with:

```js
headingDeg: interpolateHeadingDegrees(fromOffset.headingDeg, toOffset.headingDeg, eased)
```

This preserves the intended fix while preventing long-way rotations across the wrap boundary.

## QA / Validation Plan

### Static validation

Confirm the diff remains limited to `mbta/app.js` and only touches the intended utility/function areas.

### Unit-style browser-console validation for heading interpolation

After adding `interpolateHeadingDegrees`, validate edge cases in DevTools:

```js
console.assert(interpolateHeadingDegrees(350, 10, 0) === 350);
console.assert(interpolateHeadingDegrees(350, 10, 0.5) === 360);
console.assert(interpolateHeadingDegrees(350, 10, 1) === 370);
console.assert(interpolateHeadingDegrees(10, 350, 0.5) === 0);
console.assert(interpolateHeadingDegrees(45, 90, 0.5) === 67.5);
console.assert(interpolateHeadingDegrees(undefined, 90, 0.5) === 90);
console.assert(interpolateHeadingDegrees(90, undefined, 0.5) === 90);
```

CSS `rotate()` accepts values above `360deg`, so `370deg` is visually equivalent to `10deg` while preserving the shortest interpolation path.

### Runtime validation for vehicle heading

With vehicles visible:

1. Wait for a polling update that animates an existing vehicle.
2. Inspect `.vehicle-offset-marker` on that vehicle.
3. Confirm `--vehicle-heading` changes during animation.
4. Confirm the arrow does not spin the long way when crossing the wrap boundary.

### Alert validation

Select a route with alerts and confirm:

- `.route-alert`, `.alert-toggle`, and `.alert-indicator` still receive `data-lifecycle`.
- `.alert-item` no longer receives `data-lifecycle`.
- Alert colors, alert count, and show/hide behavior are unchanged.

### Popup validation

Open a stop popup and a vehicle popup and confirm:

- Travel-time rows still have `popup-info-value popup-time-value`.
- Standard info rows still have `popup-info-value`.
- Label/value text remains escaped and displayed normally.

## Conclusion

Two current diffs are safe as written: `popupInfoRow()` and `renderAlerts()`.

The heading fix is directionally correct but should be adjusted before considering the debug complete. The current direct interpolation fixes frozen headings but can introduce long-way arrow rotation across angle wrap boundaries. The next implementation step should replace direct heading interpolation with shortest-path angle interpolation.
